# goals.md — プロジェクトの目的と完成形

作成日時: 2026-07-07 07:09
更新日時: 2026-07-07 07:09

## 目的

ローカル LLM と協働で技術ブログ記事を執筆・校正するスタンドアロンのデスクトップエディタを作る。

- 技術ブログ記事（technical-notes.com 向け）の執筆を LLM と協働で行う。
- 普通のエディタ（Word 的な操作感）としても成立させる。LLM 支援は「必要なときに呼ぶ」ものであり、押し付けない。
- すべてローカルで完結する（クラウド API に依存しない。Web 検索 API のみ外部）。

## 完成形

以下がそろった状態を完成形とする。

1. **エディタ**: TipTap ベースの WYSIWYG。Markdown ネイティブ、画像挿入、ワークスペース単位の文書管理（左サイドバー）、SQLite への自動永続化。
2. **校正**: 二モードが動作する。
   - インライン校正: 文・選択範囲・単一段落 → 同一画面で diff 表示、accept/reject。
   - 分割ビュー校正: 段落を超える範囲 → 左右分割で before/after 比較、段落ごと accept/reject。
3. **RAG**: 過去記事・リファレンス・Web 取得原文を Ruri 埋め込み + hybrid search（sqlite-vec + FTS5）で検索し、執筆コンテキストに供給できる。
4. **Web 検索**: クエリ分解 → 取得 → 本文抽出 → 要約 → RAG 二層保存（原文チャンク + ソースノート）が一連で動く。出典 URL・取得日時を必ず保持する。
5. **書き出し**: Markdown として記事を書き出せる。

## 重視する価値

- **エディタとしての快適さが最優先**。LLM 機能のためにエディタの応答性・操作感を犠牲にしない。
- **疎結合**: モデル同士は直接連携させない。FastAPI が仲介し、ornith の成果は RAG 経由で Gemma 4 に渡す。
- **明示制御**: RAG / Web 検索の発火はアプリ側（ボタン・トグル）が制御する。エージェント的な自動判断や function calling には初期は依存しない。
- **出典の追跡可能性**: Web 由来の知識には必ず `source_url` と `fetched_at` を残し、記事の引用管理に使えるようにする。
- **既存資産の流用**: news-desk（dual-port 構成・`<think>` パーサ）、mem-chat（hybrid search レイヤー）を再実装せず流用する。

## 非スコープ（初期）

- クラウド同期 / マルチユーザー。
- CMS への直接投稿（WordPress MCP 連携は将来拡張として別途検討）。

## 前提環境

- Windows 11 / 48GB VRAM。
- llama.cpp（`runtime/llama.cpp/`）+ GGUF モデル（`models/`）: Gemma 4（:8080, 執筆・校正・画像）、ornith 9B（:8081, 検索・要約・reasoning）。
- 埋め込みモデル: Ruri v3 310m（`models/embeddings/` にキャッシュ済み）。

詳細仕様は [../spec.md](../spec.md) を参照。
